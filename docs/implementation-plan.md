# Zotero DataCheck 实现规划

## 1. 项目定位

Zotero DataCheck 的目标不是自动裁定论文数据造假，而是为用户提供一个在 Zotero 工作流内部即可使用的、可解释的数值异常筛查器。

项目应始终坚持以下原则：

- 输出的是风险信号，而不是结论性指控
- 每条告警都必须给出证据位置、统计依据、适用条件和局限说明
- 检测器必须可复现、可调阈值、可测试
- 数据提取与统计审查解耦，避免后续更换表格入口时重写整个引擎

## 2. 已确认的产品约束

根据当前 README，本项目的约束已经明确为：

- 第一阶段不做 OCR
- 第一阶段不依赖手动粘贴表格作为主路径，而是要求用户在 Zotero PDF 阅读器中手动框选需要分析的表格区域
- 第二阶段再做 PDF 自动表格定位与提取
- 核心分析逻辑优先用 Python 实现，TypeScript 主要负责 Zotero 插件壳层

后续实现必须以这四条为准，不再回退到“先做 CSV 粘贴为主”的旧路线。

## 3. 可行性结论：Python 后端如何落地

### 3.1 结论

Python 后端是可行的，但更合理的实现方式是“Zotero 插件前端 + 本地 Python sidecar 服务”，而不是把 Python 直接嵌入 zotero-plugin-template 的插件运行时。

### 3.2 依据

对 zotero-plugin-template 及其 README 中提到的项目做了窄范围核实后，可以得出三个判断：

- zotero-plugin-template 本身是基于 Zotero 的 JavaScript 或 TypeScript 插件沙箱构建的，核心运行时不是 Node.js 服务器环境，也没有现成的 Python 嵌入模式
- 参考项目例如 zotero-gpt，主要做法是直接调用 Zotero 内部 PDF 阅读器和 HTTP API，而不是在插件内部塞入 Python 解释器
- 这类插件如果需要更重的计算或外部能力，最稳妥的工程化方式是通过本地 HTTP 服务、文件交换或独立伴随进程来协作

### 3.3 架构决策

因此本项目建议采用以下结构：

- TypeScript 插件层：负责 Zotero 菜单入口、PDF 阅读器交互、框选信息获取、任务发起、进度展示、结果面板、结果存储
- Python 分析层：负责表格重建、数值标准化、统计检测、报告生成
- 通信层：优先采用 localhost HTTP + JSON 协议；开发期由用户手动启动 Python 服务，后续再考虑自动启动或 PyInstaller 打包

这一决策可以最大程度满足“核心逻辑用 Python 编写”的目标，同时避免在插件运行时里做高风险的跨语言嵌入。

## 4. 第一阶段用户流程

第一阶段的主流程建议固定如下：

1. 用户在 Zotero 中打开某篇论文的 PDF。
2. 用户启动插件的“框选表格并分析”命令。
3. 插件在 PDF 阅读器中进入框选模式，让用户框出表格区域。
4. 插件获取目标页码、矩形框坐标，以及该框内或邻近的 PDF 文本层与几何信息。
5. 插件将这些原始元素发送给 Python 后端。
6. Python 后端重建表格网格、标准化数值、运行检测器并返回结构化报告。
7. 插件在 Zotero 内显示报告，并允许保存、导出或回写摘要。

这个流程的关键点是：表格定位由用户先确定，算法在第一阶段只负责“区域内表格重建和异常检测”，不承担“先从整页里把表找出来”的任务。

## 5. 总体架构

### 5.1 TypeScript 插件层职责

- 注册菜单命令、右键入口和批处理入口
- 与 Zotero PDF 阅读器交互，获取当前 reader、页码、框选矩形和原始文本对象
- 调用 Python 后端并处理超时、错误和版本兼容
- 展示结果摘要、详细证据和导出入口
- 将报告保存到插件数据目录，并在 Zotero 条目层面建立索引或摘要

### 5.2 Python 分析层职责

- 接收 PDF 区域原始文本与几何信息
- 将文本对象聚类为行、列和单元格，重建二维表格
- 标准化数值、百分比、科学计数法和 p 值表达式
- 运行统计审查规则
- 输出结构化报告和诊断信息

### 5.3 存储与报告层职责

- 完整 JSON 报告保存到插件数据目录
- 给 Zotero 项目条目回写简短摘要、风险标签或链接
- 保留运行时间、参数版本和后端版本，便于复现

## 6. 第一阶段的表格获取方案

### 6.1 目标

第一阶段不是“自动找表”，而是“把用户框选的 PDF 区域稳定变成可分析表格”。

### 6.2 插件侧要获取的数据

插件在用户框选完成后，需要采集以下信息：

- Zotero 条目 ID、附件 key、文件路径或稳定标识
- 目标页码
- 框选矩形的页面坐标
- 该页 PDF 文本层中的文本项及其几何信息，例如文本内容、坐标、宽高、字体大小、旋转信息
- 若可获取，还可带上线段、边框或注释辅助信息

### 6.3 依据

模板生态中的项目已经表明，Zotero 插件可以访问 PDF 阅读器和 PDF.js 的文本内容、页面对象与几何布局信息。这意味着：

- 第一阶段完全可以不依赖 OCR
- 只要用户先框定区域，就可以对该区域内的文本对象做表格重建
- 第二阶段自动抽表也有明确技术基础，因为 PDF 文本层和版面几何本身就是可用的

### 6.4 Python 端表格重建流程

Python 端建议按以下顺序处理：

1. 根据矩形框过滤文本对象
2. 对文本对象按 y 坐标聚类形成候选行
3. 对每一行按 x 坐标排序并估计列边界
4. 合并跨列或跨行文本，生成候选单元格
5. 对重建结果生成一个中间表示，必要时返回给前端做人工确认
6. 输出最终的 TableDocument

### 6.5 第一阶段的现实边界

第一阶段只保证以下情况的成功率优先：

- 文本层存在、不是扫描图片的论文 PDF
- 表格结构相对规整，列对齐较稳定
- 用户框选足够精确，没有把正文段落和图注大面积混入

对于复杂跨页表、强旋转表、图片型表格、扫描件，第一阶段只做“识别失败并解释原因”，不追求覆盖。

## 7. 第二阶段的自动 PDF 表格提取方向

第二阶段的目标是从“用户手动框选表格”升级到“插件自动在 PDF 中定位表格并抽取”。

建议优先采用非 OCR 路线：

- 基于文本块密度、对齐关系和空白分割检测候选表格区域
- 利用相似的列 x 坐标和重复的行 y 坐标识别网格结构
- 利用线条、边框、标题关键词如 Table 1、Tab.、Supplementary Table 等辅助定位
- 对检测出的候选区域给用户一个确认界面，而不是直接静默分析

只有当论文是图片型 PDF 或扫描版，才评估 OCR 辅助模块。OCR 不应成为主线方案。

## 9. Python 服务接口设计

### 9.1 推荐协议

推荐采用本地 HTTP 服务，原因如下：

- TypeScript 插件侧发 HTTP 请求最稳定，和现有 Zotero 插件常见做法一致
- Python 端便于用 FastAPI 或 Flask 快速定义接口
- 后续如果需要把 Python 端独立打包，HTTP 接口最容易保持兼容

### 9.2 建议接口

- GET /health：检查服务是否在线、版本是否匹配
- POST /analyze-region：输入框选区域文本几何信息，输出表格重建和统计报告
- POST /reconstruct-table：只做表格重建，用于调试和人工确认
- POST /export-report：将报告转换为更适合归档的格式

### 9.3 开发与发布策略

- 开发阶段：用户本地启动 Python 服务
- 第一个可用版本：插件在找不到服务时给出明确提示，而不是尝试隐式安装
- 后续版本：再考虑自动启动或打包为单文件可执行程序

## 10. 统计检测模块设计

每个检测器都应独立实现，输入统一数据模型，输出统一结果接口。

### 10.1 精确重复与近重复检测

目标：识别不太可能自然频繁出现的重复值、重复行、重复列和近重复值。

实现方法：

- 统计单元格数值频次
- 结合数值精度判断近重复聚类
- 对行向量和列向量进行签名比对，识别整体重复或高度相似结构

### 10.2 平移、缩放与仿射关系检测

目标：识别一组数值疑似由另一组数值经过加常数、乘常数或线性变换得到。

实现方法：

- 对长度足够的行向量和列向量两两拟合 y = ax + b
- 计算残差、相关性和有效配对数量
- 在残差显著低于表格分辨率时给出告警

### 10.3 异常趋势与异常平滑检测

目标：发现过于整齐、过于平滑或符号变化异常少的序列。

实现方法：

- 统计单调运行长度和符号变化次数
- 计算一阶差分和二阶差分的离散程度
- 对候选序列做线性拟合并评估残差是否异常小

### 10.4 本福特定律检测

目标：在适用条件满足时检查首位数字分布是否异常偏离本福特定律。

实现方法：

- 判断数据是否满足适用前提
- 提取首位有效数字
- 计算 MAD、卡方统计量等偏离指标
- 不适用时明确返回原因，不输出误导性分数

### 10.5 p 值分布检测

目标：识别显著性阈值附近的异常堆积和过多接近 0.05 的 p 值。

实现方法：

- 识别 p = 0.043、p < 0.05、P=.049 等表达式
- 区分精确值和区间值
- 对精确 p 值做阈值附近的分箱统计
- 在样本足够时可加简单 p-curve 视图，但只作为辅助信息

## 11. TypeScript 与 Python 的代码分工

### 11.1 适合留在 TypeScript 插件层的部分

- Zotero API 调用
- PDF 阅读器状态和页面交互
- 框选 UI 与结果面板
- 报告缓存与回写
- 错误展示、进度反馈和用户配置

### 11.2 适合放在 Python 端的部分

- 表格重建算法
- 数值标准化
- 检测器集合
- 报告聚合和阈值配置逻辑
- 大部分单元测试和算法实验

这样划分后，真正需要你长期维护的复杂逻辑基本可以集中在 Python 中。

## 12. 测试与验证策略

### 12.1 TypeScript 侧

- 验证 PDF 阅读器中框选结果是否正确映射到页面坐标
- 验证插件与 Python 服务的协议、超时和错误提示
- 验证结果面板、导出和回写流程

### 12.2 Python 侧

- 用 pytest 测表格重建
- 用合成样例测重复、平移缩放、趋势、本福特和 p 值检测
- 用真实但匿名化的样例估计误报情况
- 对不同版式 PDF 的文本层样例做回归测试

### 12.3 关键验证指标

- 表格重建成功率
- 错位和合并单元格的错误率
- 各检测器的误报和漏报倾向
- 服务接口稳定性和结果可复现性

## 13. 里程碑建议

### M0：仓库与脚手架落地

- 将 zotero-plugin-template 迁入本仓库
- 重命名插件元信息
- 建立 Python 服务目录和最小启动脚本
- 跑通插件和 Python 服务的本地开发流程

### M1：手动框选与区域提取

- 在 Zotero PDF 阅读器中实现手动框选
- 提取目标页的文本层和几何数据
- 建立 TypeScript 到 Python 的分析请求协议

### M2：表格重建与标准化

- Python 端实现区域内表格重建
- 定义 TableDocument 和 NormalizedCell
- 实现 p 值表达式、百分号、科学计数法等标准化

### M3：首批检测器

- 实现重复与近重复检测
- 实现平移或缩放检测
- 实现异常趋势、本福特和 p 值分布检测

### M4：报告与 Zotero 集成

- 实现结果面板和摘要视图
- 支持导出 JSON 或 Markdown
- 支持把摘要写回文献笔记或内部索引

### M5：自动 PDF 表格提取

- 建立整页候选表格检测
- 给出候选区域预览和人工确认
- 将自动提取接入现有分析流水线

## 14. 主要风险与对应策略

### 14.1 插件无法直接优雅托管 Python 运行时

策略：第一版默认把 Python 作为独立本地服务，先解决功能闭环；安装体验在后续打磨。

### 14.2 PDF 版式复杂，区域内也可能难以重建表格

策略：先聚焦规整论文表格；对失败场景返回明确诊断，而不是输出错误结果。

### 14.3 统计检测天然存在误报

策略：所有结果默认用“风险提示”文案，不做自动定性；前端必须展示局限说明。

### 14.4 第二阶段自动抽表可能比预期更难

策略：第一阶段的数据契约和检测引擎要与“表格定位方式”解耦，这样即使自动抽表推进较慢，手动框选路径也能独立持续可用。

## 15. 当前建议的下一步

如果现在进入编码阶段，建议按以下顺序推进：

1. 将 zotero-plugin-template 迁入 zotero-datacheck，并完成插件名称、ID 和首选项前缀重命名。
2. 在插件层实现一个最小命令，能够从 PDF 阅读器拿到当前页和手动框选矩形。
3. 新建 Python 服务目录，先做 /health 和 /reconstruct-table 两个最小接口。
4. 用一两个真实论文表格样例打通“框选 -> 文本层提取 -> 表格重建 -> JSON 返回”。
5. 在这个最小闭环稳定后，再加统计检测器。

这个顺序符合当前 README 的约束，也最能控制技术风险。
